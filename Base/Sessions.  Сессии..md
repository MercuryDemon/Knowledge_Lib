# Сессии

Это подход к реализации взаимодействия клиета и Stateful-сервиса. см. [[Stateful vs stateless wed-services design]]


## Как это работает

1. ПОльзователь отправляет логин и пароль
2. Сервер проверяет данные в БД (есть ли такой пользователь)
3. Сервер создает сессию на своей стороне и генерирует cookie, в которой передает ID сессии
4. Пользователь отправляет эту cookie при каждом запросе
5. Сервер валидирует ее и дает юзеру доступ на основании сессии
6. когда пользователь выходит из системы, сервер удаляет сессию и чистит cookie.


## Особенности:

- сессии хранятся в БД или в кэше
- каждый запрос пользователя - это каждый раз поход в базу или кэш для проверки валидности сессии
- данные переданные в куках зашифрованы - данные непрозрачны
- используются в основном в SSR - server side rendering приложениях


## HTTP Cookie

Это http-header. который хранит:
- ID сессии
- пользовательские настройки
- метррики (отслеживает действия пользователя)

Состоит из: **имени**, **значения**, а также **атрибутов** и **флагов** (опционально)

Устанавливается сервером с помощью заголовка  Set-Cookie

## Плюсы

- Проще контролировать валидность подключений. во избежании утери или захвата данных пользователя злоумышленниками.
