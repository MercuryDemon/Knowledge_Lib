# Архитектура

-  это фундаментальная организация системы, воплощенная в её компонентах, их отношениях друг с другом и с окружением, а также принципы, определяющие её проектирование и эволюцию.

# Цели

- Масштабируемость
- Поддерживаемость
- Тестируемость
- Надежность
- Безопасность
- Производительность

# Стили архитектуры

## Монолит

- всё приложение - один большой блок.

плюсы:
- Простота разработки
- Легко деплоить
- Простые транзакции

Минусы:
- Сложно масштабировать
- Технологический лок
- Ошибка валит всё

## Микросервисы

- Много маленьких независимых сервисов.

Плюсы:
- Масштабируются независимо
- Разные технологии
- Изоляция ошибок

Минусы:
- Сложность сети
- Распределенные транзакции
- Труднее дебажить


## Слоёная Архитектура

- Разделение сервиса на слои 

# Архитектурные паттерны

## Outbox

Transactional outbox.

1. Какой-то сервис генерирует сообщение
2. атомарно записывает его в БД (транзакция)
3. отдельный воркер этого сервиса достает это сообщение из бд и отправляет в кафку, проверяет, что сообщение получено и удаляет или помечает флагом как отправленное сообщение.

- реализует принцип at-least-once


## Inbox 

- Суть в том, чтобы использовать базу данных на входе сообщений, как лог входящих.

1. пишем входящее сообщение в базу в рамках той же транзакции, что обрабатывает начальные бизнес данные.
2. запись в базе имеет уникальное поле message_id
3. перед тем как обработать входящее сообщения, сервис консюмер проверяет Inbox таблицу БД, чтобы узнать, было ли сообщение с этим идентификатором уже обработано.
4. Если это дубликат, сообщение подтверждается и игнорируется.

- паттерн выступает фильтром дубликатов
- убирает проблему блокирования очереди, изза необработанного сообщения

минусы:
- оверхэд по производительности
- сложность реализации
- БД может стать узким местом, при большом потоке сообщений
- оверхэд по памяти (т.к. надо хранить сообщения)

## DLQ

Dead letter Queue - очередь мертвых сообщений.  Хранит сообщения, которые не удалось обработать в основной очереди.

1. Сообщение попадает в основную очередь
2. Консюмер пытается его обработать
3. Если ошибка временная → сообщение возвращается в очередь (re-queue) или идет в retry-очередь
4. После N неудачных попыток → сообщение перемещается в DLQ
5. Инженеры анализируют DLQ и принимают решение

## SAGA

- способ управления соглаосванностью данных в распределенных системах

Нам надо чтобы несколько микросервисов завершили все свои внутренние транзакции последовательно в рамкох общей транзакции. Если гдето чтото не отработает - вся цепочка откатится.

- реализуется через компенсирующие тарнзакции

### Хореография:

- друг за другом в обратном порядке движения транзакции.

![[Pasted image 20260317134324.png]]

в случае, если в одном сервисе транзакция не обрабатывается этот сервис отправляет в бровер event отмены, который считывается в обратном порядке предыдущим микром, этот микросервис применяет компенсирующую транзакцию и создает свой event отмены и так по цепочке до сервиса, который начал общую транзакцию

![[Pasted image 20260317134810.png]]


плюсы: 
- подходит для простой бизнес логики
- не требует реализации доп сервисов
- не вводит единую точку отказа, поскольку ответственно сть распределена между всеми учасниками SAGA


минусы:
- при наличии сложной бизнеслогики, трудно управлять внутренними транзакциями (можно чтото упустить) и интегрировать новые сервисы в эту систему.
- есть риск циклической зависимости между учасниками, так как им приходится потреблять команды друг друга
- Сложное интеграционное тестирование, т.к. все службы должны быть запущены для иммитации транзакции.


### Оркестрация

Один главный сервис, который координирует другие, компенсационные транзакции выполняются одновременно

- требует поднятия отдельного сервиса, либо добавления в какой либо из существующих модуля SAGA

![[Pasted image 20260317135935.png]]



плюсы:
- нагляднее логика взаимодействия
- проще добавлять новый сервис
- логика вынесена в одно место, не нужно в каждом микре настраивать сложное взаимодействие с брокером.
- проще дебажить и тестировать
- подходит для запутанной бизнеслогики

Минусы:
- есть потенциальная точка отказа микр(САГА)
- Нужно писать отдельный микросервис, что долго и сложно
