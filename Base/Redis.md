- есть много других структур хранения данных

# Где и когда применять

В основном используется для кэширования. Кэш - буквально отдельно стоящая БД которая хранит копии популярных данных из основной БД **очень быстро работает на чтение** данных, однако **не обеспечивает консистентности и надёжности** данных. (см. [[Кэширование]])


Если в нашем сервисе есть популярные(часто запрашиваемые) данные, важна скорость отдачи этих данных и не нужна консистентность - **имплементируем КЭШ через Redis**.
# Особенности

- Хранит данные парами Key-Value
- Быстрый поиск работает по конкретному ключу, если есть часть ключаили ключа нет, данные не достать
- достает всегда все данные по ключу
- нет отношений между кусками данных, нет сложных запросов или фильтров
- синхронизация данных в кластере не детерминирована по времени - нет консистентности данных.

# Фичи в Redis

## Redis: leaderboard

- Даем баллы пользователям за активность, повышая их в общем рейтинге.
![[Pasted image 20260114221456.png]]
leaderboard реализовыан через структуру данных ZSET - сортированное множество, которое автосатически сортирует данные в зависимости от значения.

## Репликация Redis

- Обьединяем все серверы Redis в кластер, они синхронизируют данные, и равномерно распределяем между ними нагрузку.
- повышаем отказоустойчивость.


## Масштабирование Redis

- Настраиваем кластер так, чтобы данные хранились распределенно(часть на одном сервере, часть на другом), редис из коробки позволяет это сделать. Он сам будет перенаправлять запросы на нужную машину.

## Механизмы синхронизации

- Есть поддержка атомарных операций,
- есть Redis transaction
- optimistic/pessimistic lock (mutex)
