# Сервер  и клиенты

- Задача приложения – выставить HTTP-страницу со своими метриками в  определенном формате.

- Сервер периодически делает HTTP-запрос GET /metrics к нашему приложению — это называется **scrape**. Интервал запроса может быть любым, можно настроить endpoint и заменить HTTP на что-то другое. В примерах дальше будем считать, что scrape делается раз в 30 секунд.

- Ответ приложения сервер сохраняет в БД с текущим timestamp.

- Метрики приложения остаются в памяти приложения. Мы их дополняем и агрегируем дальше.


### Безопасность

способы закрыть доступ к метрикам ото всех кроме нас:

- настроить аутентификацию в Prometheus, чтобы он сам нужные заголовки отправлял при скрэйпинге.
- настроить эндпоинт на отдельный порт который не торчит
- настроить файерволл(ну это пиздей заёб)
- сделать whitelist на уровне приложения (как РКН на уровне интернета)

## Формат данных

Метрики имеют человеко-читаемый формат

пример:
```
# HELP http_requests_total Requests made to public API
# TYPE http_requests_total counter
http_requests_total{method="POST", url="/messages"} 1
http_requests_total{method="GET", url="/messages"} 3
http_requests_total{method="POST", url="/login"} 2
```

где:
- HELP — описание для помощи человекам;
- TYPE — тип метрики;
-  http_requests_total — имя метрики;
- набор key-value лейблов (можно еще называть их тегами) (пример: method = "GET");
- значение метрики (64-bit float aka double);
- после сбора в БД добавляется еще timestamp.

фактически имя метрики и лэйблы - это название таблицы, которая хранит:
- 1 значение - timestamp
- 2 значение - float64 

# Кардинальность

Каждое новое значение лейбла это новый временной ряд, по этому мы заранее должны понимать как много их моджет быть (их должно быть немного).

напрмер значений для лейбла "Method" всего 4:
- GET
- POST
- PUT
- DELETE

Например, мы пишем метрики о HTTP-запросах. Перемножаем все возможные значения всех лейблов: 2 HTTP глагола, 7 урлов, 5 реплик сервиса, 3 вида ответов (2xx, 3xx, 4xx), 4 браузера. 840 временных рядов! Ну то есть это как 840 таблиц в sql. Prometheus справляется с десятками миллионов рядов, но комбинаторный взрыв можно устроить очень быстро.

# Среднее арифметическое и Медиана

Говоря о метриках мы работаем с временными рядами (timestamp - value), соответсвенно убрав ключи у нас остается просто [[Массив (array)|массив]] чисел
## Среднее арифметическое

Это сумма всех элементов, деленная на их количество

## Медиана

 – это «средний элемент», то есть буквально в середине массива (если его упорядочить). Чтобы ее найти:

1. Упорядочим массив (отсортируем по значению)
2. Если длина массива нечетная, берем элемент посередине
3. Если длина массива четная, берем два элемента посередине и считаем их среднее арифметическое

- половина элементов массива больше либо равна ей, а другая половина элементов — меньше либо равна.
# Перцентили

- N-й перцентиль - это такое число, что N% элементов массива меньше или равны этому числу
- 95 перцинтиль = 95% чисел в массиве меньше или равны этому числу

- Перцентили позволяют нарезать слоями ваш массив и понять, на какие группы делятся элементы: в среднем они такие-то, в основном такие-то, в подавляющем большинстве такие-то и т.д

# Семплирование

Для метрик используются не все данные за всё время, т.к. их дорого хранить и долго упорядочивать и обсчитывать. Используется срез данных вовремени.

Храним например 100 элементов, а когда место закончится решим какой почикать и запишем на его место новый:
- Sliding Window - выбрасываем самый2 старый элемент
- Forward Decay - выбираем элемент рандомно, но чем он старее тем больше вероятность, что выкинем его.

# PromQL

## Запросы

Каждая метрика — это временной ряд, можно сказать, отдельная таблица. 
Имя таблицы — это набор лейблов, а значения — одно число, записанное в разные моменты времени: `[(day1, t1), (day2, t2), ...]`. 
Запросы могут:
1. достать значения из конкретного ряда за нужное время
2. то же самое сразу из нескольких рядов
3. сгруппировать или провести вычисления над результатом

## Простые типы 

- Строки — чтобы запрашивать лейблы и их значения.
- Числа — только double, они могут участвовать в функциях.
## Типы выражений и API

### Instant vector

Мгновенный вектор. Тип данных содержащий в себе набор временных рядов (key = лейблы) с одним значением (value) в один момент времени (timestamp). 
- Это моментальныйснимок состояния системы. 

Применяется:
- мониторинг в реальном времени
- алертинг(типа как датчик, значение всегда какое-то не 0, но если 0 = что-то не так)

пример запроса возвращающего instant vector:
```PromQl
http_requests_total{job="prometheus",group="canary"}
```
### Range vector

Набор временных рядов, каждый из которых содержит группу значений (семплов) за определенный промежуток времени, предшестующий моменту запроса.

применяется:
- рассчет скорости изменения: Вычисление количества запросов в секунду с помощью функций `rate()` или `irate()`
- Анализ трендов: Выявление постепенного роста потребления памяти или количества ошибок
- Агрегация за период: Расчет среднего, максимального или перцентильного значения за последние 10 минут с помощью функций `avg_over_time()`, `max_over_time()` и т.д.
- Обнаружение аномалий: Поиск резких скачков или падений метрики

кроме того: 
- Функции, как rate(), irate(), increase(), avg_over_time(), принимают на вход range vector, а возвращают instant vector с результатом вычислений . Это позволяет строить графики

пример запроса возвращающего range vector:
```PromQl
http_requests_total{job="prometheus",group="canary"}[5m]
```
- в квадратных скобках указано время (5 минут). Дивапазон от данного момента на 5 минут в прошлое, и значения за этот период.

На что влияет число в квадратных скобках: в каком-то смысле, благодаря ему можно тюнить степень «сглаживания» или «разрешение» (точность).

- больше интервал — более гладкий результат, видно тренд (общую картину);
- меньше интервал — более шумный результат, видно пики и резкие скачки;\
- слишком малый интервал и не захватили минимум две точки — нет результата.


Range vector чтобы вывести на график, нужно агрегировать(обработать) в instant vector
